Teste é parte do processo e reduz riscos de falhas em produção.

Objetivos

O que você vai dominar hoje

  • Diferenciar níveis de teste: unitário, integração, sistema e aceitação.
  • Escrever casos de teste a partir de requisitos e critérios de aceitação.
  • Entender defeito, falha e causa.
  • Definir uma estratégia mínima de teste para um projeto pequeno.
Conceitos

O que é testar?

  • Teste é executar e avaliar para encontrar problemas e confirmar comportamento.
  • Defeito é um erro no artefato (requisito, código, design).
  • Falha é o comportamento incorreto observado em execução.

Testar não prova ausência de defeitos; aumenta confiança e reduz risco.

Níveis

Onde o teste acontece?

Nível Foco Exemplo
Unitário Funções/métodos isolados Calcular data de lembrete
Integração Integração entre módulos API + banco + autenticação
Sistema Sistema completo Fluxo: criar tarefa → alertar
Aceitação Critérios do usuário/cliente Validação com professor/aluno

Quanto mais cedo o defeito é encontrado, menor o custo para corrigir.

Casos de teste

Como escrever um caso de teste

  • Pré-condições: estado inicial (ex.: usuário logado).
  • Entradas: dados fornecidos (ex.: tarefa e data).
  • Passos: ações do usuário/sistema.
  • Resultado esperado: o que deve acontecer.

Casos de teste devem ser reprodutíveis e ligados a um requisito.

Atividade

Defina testes para requisitos

Em grupo (10–15 min)

  • Escolha 2 requisitos do backlog do “app organizador”.
  • Para cada requisito, escreva 2 casos de teste.
  • Inclua 1 caso de erro/limite (ex.: data inválida, campo vazio).

Use critérios de aceitação para definir o resultado esperado.
