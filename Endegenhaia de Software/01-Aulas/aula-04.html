Requisitos são a base para reduzir retrabalho e alinhar expectativa.

Objetivos

O que você vai dominar hoje

  • Diferenciar requisitos funcionais e não funcionais.
  • Aplicar um fluxo: elicitar → analisar → especificar → validar.
  • Escrever requisitos testáveis com critérios de aceitação.
  • Reconhecer riscos: ambiguidade, conflito, escopo invisível.
Conceito

Tipos de requisitos

Funcionais

O que o sistema faz: funcionalidades e regras.

Ex.: cadastrar, buscar, calcular, emitir relatório.

Não funcionais

Como o sistema deve ser: qualidade e restrições.

Ex.: desempenho, segurança, usabilidade, disponibilidade.

Requisito bom é claro, necessário, consistente e verificável.

Processo

Um roteiro prático

Etapa O que fazer Saídas
Elicitação Entrevistas, observação, análise de documentos Lista inicial de necessidades
Análise Conflitos, prioridades, regras e dependências Requisitos refinados
Especificação Documentar de forma testável Backlog, casos de uso, SRS
Validação Revisões e aceitação com stakeholders Versão aprovada
Ferramentas

Formatos de especificação (práticos)

História de usuário

Como <tipo>, eu quero <objetivo> para <benefício>.

+ critérios de aceitação

Caso de uso

Ator, objetivo, fluxo principal, alternativas, exceções.

Bom para regras e integrações

O importante é ser compreensível e permitir validar o requisito com testes.

Atividade

Escreva requisitos com aceitação

Em grupo (10–15 min)

  • Escreva 3 requisitos funcionais do “app organizador”.
  • Escreva 2 requisitos não funcionais (segurança, usabilidade, desempenho).
  • Para cada requisito funcional, crie 2 critérios de aceitação.

Critério de aceitação deve ser observável: alguém consegue testar e dizer “passou/falhou”.
